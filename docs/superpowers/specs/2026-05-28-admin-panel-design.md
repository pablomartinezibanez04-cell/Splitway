# Splitway Admin Panel — Design Document

**Status:** Approved
**Date:** 2026-05-28
**Author:** PabloMartinezIbanez

## 1. Context and Goal

Splitway is a Flutter mobile app for GPS-based route tracking, timed sessions, free rides, and drag-strip-style speed measurements. It uses Supabase as backend (Postgres + Auth + Storage + Edge Functions) with strict per-user Row-Level Security.

We need an **admin web panel** so the development team can:

- Manage users (view, ban, reset password, delete)
- Manage routes and sessions (view, edit, delete, mark official)
- Monitor app health via logs and usage metrics

The panel is for **internal team use** (≤3 people initially), not end-user-facing.

## 2. Scope

### In scope

- Self-hosted Next.js web app inside `admin/` directory of this monorepo
- Auth via Supabase with two admin roles: `admin` and `superadmin`
- Full CRUD over users, routes, sessions
- Dashboard with KPIs and charts
- Diagnostic log viewer with filters and live-tail
- Audit log of all admin actions

### Out of scope (YAGNI)

- Realtime live updates (polling is sufficient)
- Bulk actions
- CSV export (add later if needed)
- 2FA/MFA for admins
- UI dark/light mode toggle
- Internationalization (Spanish only)
- End-user-facing admin UI inside the Flutter app

## 4. Authentication and Authorization

### Roles

New column on `profiles`:

```sql
alter table public.profiles
  add column role text not null default 'user'
    check (role in ('user', 'admin', 'superadmin'));

create index profiles_role_idx on public.profiles(role) where role != 'user';
```

Role semantics:

- **`user`**: regular Flutter app user. Cannot access the admin panel.
- **`admin`**: read-only across the platform + moderation actions (ban, reset password, edit route metadata, delete sessions, mark routes as official).
- **`superadmin`**: everything `admin` can do + delete users + grant/revoke admin roles to others.

### Bootstrap

One-time seed migration promotes the founding superadmin:

```sql
update public.profiles
set role = 'superadmin'
where id = (select id from auth.users where email = 'pabmariba@gmail.com');
```

Subsequent admins are granted via the `/settings` page (only superadmins can do this).

### Login flow

1. User navigates to `/login`, signs in with email + password via Supabase Auth.
2. `middleware.ts` intercepts every request except `/login` and static assets:
   - No session → redirect to `/login`
   - Session exists but `profiles.role` is not `admin` or `superadmin` → sign out, redirect to `/login?error=forbidden`
3. Server Components and Server Actions additionally call `requireAdmin()` / `requireSuperadmin()` for defense-in-depth.

### Service role handling

- `lib/supabase/admin.ts` imports `'server-only'` and is the **only** place where the service role key is read.
- All cross-user data access (reading other users' sessions, deleting users, etc.) goes through this client, called from Server Components or Server Actions.
- The service role key is never sent to the browser.

### Audit log

New table `admin_audit_log` records every mutating admin action:

```sql
create table public.admin_audit_log (
  id          uuid primary key default gen_random_uuid(),
  admin_id    uuid not null references auth.users(id),
  action      text not null,        -- 'ban_user', 'delete_route', etc.
  target_type text not null,        -- 'user', 'route', 'session', 'free_ride'
  target_id   text not null,
  details     jsonb,
  created_at  timestamptz not null default now()
);

create index admin_audit_log_created_idx on admin_audit_log(created_at desc);
create index admin_audit_log_admin_idx on admin_audit_log(admin_id, created_at desc);

alter table admin_audit_log enable row level security;

create policy "admins read audit log" on admin_audit_log for select
  using (exists (
    select 1 from profiles
    where id = auth.uid() and role in ('admin', 'superadmin')
  ));
```

Every Server Action that mutates state calls `writeAuditLog(...)` after the change succeeds.

## 5. Cross-User Data Access

The existing RLS policies restrict each user to their own data. Admins need to see everyone. The approach chosen:

**All cross-user reads/writes go through the `service_role` client on the server**, gated by `requireAdmin()` in TypeScript before any query is made.

Trade-offs considered:

- **Adding `is_admin` clauses to every RLS policy**: rejected — adds ~40 new policies across 10+ tables, lots of churn and easy to miss.
- **JWT custom claims hook**: not adopted initially; revisit if we ever need realtime subscriptions from the browser with admin scope.

This keeps existing RLS untouched and concentrates the trust boundary in two places: `requireAdmin()` and the `admin.ts` client.

## 6. Features by Section

### 6.1 Dashboard (`/`)

KPI cards:

- **Total users**: `count(*) from profiles`
- **Active users last 7 days**: distinct users who have at least one row in `session_runs`, `free_rides`, `speed_sessions`, or `app_logs` with `created_at` (or `timestamp` for logs) within the last 7 days
- **Sessions created last 7 days**: sum of rows in `session_runs` + `free_rides` + `speed_sessions` with `created_at` (or `started_at` where `created_at` is absent) within the last 7 days
- **Errors last 24 hours**: `count(*) from app_logs where level = 'error' and timestamp > now() - interval '24 hours'`
- **Storage used**: queried from Supabase Storage admin API across `avatars`, `vehicle-photos`, `route-thumbnails`. Cached for 5 minutes to avoid recomputing on every page load.

Charts:

- Line chart: signups per day, last 30 days
- Stacked line chart: session_runs + free_rides + speed_sessions per day, last 30 days
- Bar chart: top 10 routes by session count
- Pie chart: log level distribution last 24 hours

Implementation: server-side aggregation queries from Server Components. SQL views added if any query proves slow.

### 6.2 Users (`/users`)

**List** (`/users`):
- Columns: avatar, nickname, email, role, signup date, last activity, sessions count, routes count, status (active/banned)
- Filters: role, status, free-text search on email/nickname
- Server-side pagination, sorting on all columns

**Detail** (`/users/[id]`) — tabs:
- **Profile**: editable fields (nickname, bio); `role` field editable only by superadmin
- **Activity**: list of session_runs, free_rides, speed_sessions
- **Garage**: vehicles
- **Routes**: routes created by the user
- **Logs**: app_logs filtered by `user_id`
- **Danger zone** (bottom cards):
  - Ban / unban (admin+) — uses `auth.admin.updateUserById({ ban_duration: 'Xh' | 'none' })`
  - Reset password (admin+) — sends a password-reset email
  - Delete user (superadmin only) — invokes the existing `delete-user` edge function

### 6.3 Routes (`/routes`)

**List**: thumbnail, name, owner nickname, difficulty, sector count, sessions count, `location_label`, creation date. Filters by difficulty, owner, free-text search.

**Detail** (`/routes/[id]`):
- Map with the route drawn (Mapbox GL JS, uses `MAPBOX_PUBLIC_TOKEN`)
- Editable metadata: name, description, difficulty, `location_label`
- Sector list
- Sessions that used this route
- Toggle **"Official route"** — new column `route_templates.is_official boolean not null default false`. Official routes are visible to all users in the Flutter app (Flutter wiring is a separate follow-up PR).
- Actions: edit, duplicate as official, delete (confirmation modal)

### 6.4 Sessions (`/sessions`)

Three tabs:
- **Session runs** (timed route sessions)
- **Free rides**
- **Speed sessions** (drag strip)

Each tab is a table with: user, date, duration, distance, vehicle, route (when applicable), max/avg speed. Filters by user, date range, vehicle.

**Detail**:
- Map with the recorded trace (telemetry_points or free_ride_telemetry)
- Charts: speed vs time, altitude vs distance
- Sector splits if applicable
- Action: delete

### 6.5 Logs (`/logs`)

Sentry-style log viewer on `public.app_logs`:

- Filters: level (debug/info/warning/error), tag, user_id, app_version, platform, date range, free-text search in `message`
- Virtualized table (TanStack Virtual) — can hold thousands of rows efficiently
- Click a row → side panel with `stack_trace`, `context` (collapsible JSON tree), `device_model`
- **Live tail** toggle: polls every 5 seconds for new logs and prepends them

Existing indexes `(level, timestamp desc)` and `(user_id, timestamp desc)` are leveraged. Pagination is server-side.

### 6.6 Settings (`/settings`)

- **Admins**: list of users with role `admin` or `superadmin`. Promote/demote actions (superadmin only).
- **Audit log**: paginated table of `admin_audit_log` with filters by admin, action, date.
- **My account**: change password.

## 7. Server Action Pattern

All mutations follow this template:

```ts
// app/(dashboard)/users/[id]/actions.ts
'use server'
import 'server-only'
import { requireAdmin } from '@/lib/auth'
import { adminClient } from '@/lib/supabase/admin'
import { writeAuditLog } from '@/lib/audit'
import { z } from 'zod'

const banSchema = z.object({
  userId: z.string().uuid(),
  durationHours: z.number().int().positive(),
})

export async function banUser(input: z.infer<typeof banSchema>) {
  const admin = await requireAdmin()
  const { userId, durationHours } = banSchema.parse(input)

  const supa = adminClient()
  const { error } = await supa.auth.admin.updateUserById(userId, {
    ban_duration: `${durationHours}h`,
  })
  if (error) return { ok: false as const, error: error.message }

  await writeAuditLog({
    adminId: admin.id,
    action: 'ban_user',
    targetType: 'user',
    targetId: userId,
    details: { durationHours },
  })

  return { ok: true as const }
}
```

All Server Actions return `{ ok: true, data? }` or `{ ok: false, error: string }`. Errors are mapped to human-readable messages before reaching the client. `useFormState` + `sonner` toasts surface them in the UI.

## 9. Testing Strategy

- **TypeScript strict mode** with `noUncheckedIndexedAccess`
- **Generated Supabase types** via `supabase gen types typescript`
- **Zod schemas** in every Server Action
- **Vitest** for pure helpers (formatters, metric reducers, validators) — no UI or mocked-Supabase tests
- **Playwright** for critical flows only:
  - Login as admin → dashboard renders
  - Login as `user` role → rejected
  - Ban user → audit log row appears
  - Delete user (superadmin only) → permission gate works

E2E tests run against `supabase start` with seed data, never against production.

## 11. Non-Goals and Future Work

- **Realtime subscriptions**: polling is enough; revisit if log volume grows large.
- **JWT custom claims for admin role**: postponed; only needed if we want admin-scope realtime in the browser.
- **CSV export**: add a per-table export action if/when requested.
- **2FA/MFA**: Supabase supports TOTP; not urgent for a 3-person team behind email-password.
- **Flutter app changes**: this spec does not modify the Flutter app. Surfacing `is_official` routes in the app is a separate follow-up.
